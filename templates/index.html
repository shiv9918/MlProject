Pipeline Overview
Under the hood, this project follows a production-style ML architecture: data ingestion, transformation, model training with hyperparameter tuning, and a dedicated prediction pipeline consuming serialized artifacts.
Step 1 · Data Ingestion
Train / Test Split & Artifact Storage
Student performance data is loaded from CSV, split into training and test sets, and stored under the artifacts/ directory. This ensures reproducibility and a clear separation between raw and processed data.
Step 2 · Transformation
Preprocessing with ColumnTransformer
Numerical features (reading & writing scores) are standardized, while categorical features such as gender, lunch, and test preparation course are one-hot encoded. A single preprocessor.pkl object keeps this logic consistent across training and inference.
Step 3 · Model Training
Multiple Models & Hyperparameter Tuning
Algorithms like Random Forest, Gradient Boosting, XGBoost, CatBoost, and Linear Regression are evaluated. GridSearchCV tunes hyperparameters, and the best model based on test R² is persisted as model.pkl.
Step 4 · Predict Pipeline
CustomData & PredictPipeline Abstractions
User inputs from the UI are wrapped in a CustomData class, converted into a pandas DataFrame, passed through the loaded preprocessor, and then into the trained model via PredictPipeline for clean, production-like predictions.
Step 5 · Evaluation
Train/Test R² & Overfitting Checks
R² scores are computed on both training and test data to ensure the model generalizes well. The project is designed to highlight detection of overfitting and the impact of model choice and hyperparameters on performance.
Step 6 · UI Integration
Flask Frontend for Real Inputs
The prediction form collects all required features from end users and connects directly to the backend pipeline. This creates a realistic simulation of how an ML model would be consumed in a production web application.
